# 随销统计账期规则

## 变量定义

| 变量 | 含义 | 示例 |
|---|---|---|
| `T` | 统计日账期，格式 `yyyyMMdd` | `20260702` |
| `T_DATE` | `T` 对应的日期格式，格式 `yyyy-MM-dd` | `2026-07-02` |
| `T_PLUS_1` | `T+1` 日，格式 `yyyyMMdd` | `20260703` |
| `M` | `T` 所属月份，格式 `yyyyMM` | `202607` |
| `M_MINUS_1` | `M` 的上月，格式 `yyyyMM` | `202606` |
| `INTERN_CUTOFF` | 实习满 3 个月 cutoff 日期，即 `T_DATE - 3个月` | `2026-04-02` 或按报表口径取 `2026-04-01` |

核心结论：不要所有模块统一取 `MAX(p_acct_day)`，也不要把文件生成日期当作数据账期。日地市统计需要按模块分别确定账期和分区。

## 分模块取数规则

| 模块 | 业务/账期字段 | 分区字段 | 通用规则 | 说明 |
|---|---|---|---|---|
| 业务发展量 | `new_date = T_DATE` | `p_acct_day = T` | 业务日期和业务分区都取 `T` | 对应移动、宽带、FTTR、FTTR-B、151、智屏等发展量 |
| 积分 | `acct_day = T` | `p_acct_day = T_PLUS_1` | 积分账期取 `T`，积分落表分区取 `T+1` | 积分是次日分区落表 |
| 人员 | `month_id/month = M` | `p_acct_day = T` | 人员月份取 `M`，人员分区取 `T` | 人员没有日业务账期字段，主要看月份和分区 |
| 岗位/白表 | `month_id = M_MINUS_1` | `p_acct_day = T` | 岗位月取上月，分区跟人员日账期 `T` | `wb_ps_msg_zw` 比装维人员月份少 1 个月 |

一句话：

```text
业务和人员用 T 分区；
积分用 T 账期，但分区用 T+1；
岗位/白表月份用 M-1，分区用 T。
```

## 文件名日期理解

类似 `随销统计0702账期-20260706.xlsx`：

- `0702账期`：表示统计日账期 `T`。
- `20260706`：更像文件生成日期或导出日期，不应直接作为业务、人员、积分的统一分区。

判断口径时优先看文件中的“账期”，不要把导出日期套到所有表的 `p_acct_day`。

## 业务发展量规则

业务表：

```sql
dwd.dwd_d_mss_hr_dwv_d_smr_user_list_result_merge
```

日账期 `T` 应使用：

```sql
s.new_month = M
AND s.new_date = T_DATE
AND s.p_acct_day = T
```

装维业务口径：

```sql
s.dvlp_staff_id = zw.staff_id
AND s.is_zw = 1
AND zw.attr_value IN ('10', '20', '50', '72', '73', '200')
```

注意：

- 业务装维量不要用 `s.dvlp_staff_num = zw.staff_num` 关联。
- FTTR-H 使用 `qwzn_fttr`，不是 `fttr_yf`。
- 全量业务不需要关联装维人员。
- 不要用业务表后续分区替代 `T`，后续分区可能累积或修正 151、FTTR-B 等指标。

## 积分规则

积分表：

```sql
dwd.dwd_d_mss_hr_dwv_d_hrt_zw_point_zw_merge
```

日账期 `T` 应使用：

```sql
pt.acct_day = T
AND pt.p_acct_day = T_PLUS_1
```

装维人员过滤参考：

```sql
pt.staff_num = zw.staff_num
AND pt.attr_value IS NOT NULL
AND zw.attr_value IN ('10', '20', '50', '72', '73', '200')
AND (
  COALESCE(TRY_CAST(pt.fz_point AS DECIMAL(18, 2)), 0) != 0
  OR COALESCE(TRY_CAST(pt.yy_point AS DECIMAL(18, 2)), 0) != 0
)
```

积分展示规则：

```text
按当前 Excel 下半表反推，fz_point / yy_point 是先对积分明细单条四舍五入到整数，
再按地市/全省汇总；sum_point = fzPoint + yyPoint。

也就是说，不是简单地先按地市汇总原始小数后再取整。
```

注意：

- 积分账期和积分分区不是同一天。
- 当前验证口径是 `acct_day = T`、`p_acct_day = T+1`。
- 不要用 `MAX(p_acct_day)` 替代 `T+1`，后续分区可能会变动或覆盖。
- 不要把积分过滤简化成只有 `zw.attr_value IN ('10','20','50','72','73','200')`。
- 当前复核发现：仅保留 `zw.attr_value` 过滤会让全省积分差异变大；仍需要保留积分表 `pt.attr_value IS NOT NULL`、积分非零等隐含有效性过滤。
- 如后续拿到正式积分侧 SQL，应优先替换本节“装维人员过滤参考”，但 `acct_day=T`、`p_acct_day=T+1` 和“明细单条取整后汇总”的规则保持不变。

已复核示例：

```text
T=20260702
acct_day=20260702
p_acct_day=20260703

若按地市汇总原始小数后取整：
  fzPoint=47106, sumPoint=62918
报表：
  fzPoint=47101, sumPoint=62913
差异：
  约 5 分

若按积分明细单条取整后汇总：
  fzPoint=47101, yyPoint=15812, sumPoint=62913
报表：
  fzPoint=47101, yyPoint=15812, sumPoint=62913
结论：
  可对齐下半表积分。

若仅按 zw.attr_value 过滤：
  fzPoint=47165, sumPoint=62977
差异扩大：
  约 64 分
```

当前结论：

```text
暂时没有积分侧原始 SQL，但已从明细反推出关键展示口径：
积分账期仍是 acct_day=T、p_acct_day=T+1；
积分过滤仍保留 zw_type、pt.attr_value 非空、非零积分、有效装维人员和 attr_value 范围；
积分展示为明细单条取整后汇总。
后续若拿到正式积分侧 SQL，应重点确认是否在积分侧生成明细时已做 ROUND。
```

明细反推建议：

1. 固定积分账期规则：

   ```sql
   pt.acct_day = T
   AND pt.p_acct_day = T_PLUS_1
   ```

2. 固定当前有效过滤：

   ```sql
   pt.staff_num = zw.staff_num
   AND pt.attr_value IS NOT NULL
   AND zw.is_off = '1'
   AND e.zw_type IS NOT NULL
   AND zw.attr_value IN ('10', '20', '50', '72', '73', '200')
   AND (
     COALESCE(TRY_CAST(pt.fz_point AS DECIMAL(18, 2)), 0) != 0
     OR COALESCE(TRY_CAST(pt.yy_point AS DECIMAL(18, 2)), 0) != 0
   )
   ```

3. 按地市拉出积分明细，优先查看仍有差异的地市和人员。
4. 对比候选记录是否存在以下特征：
   - 特殊 `pt.attr_value`
   - 人员状态边界
   - 积分为小额或修正值
   - 同一人员多条记录
   - 地市归属与人员维表归属不一致
5. 在未拿到积分侧原始 SQL 前，不要为了消除小额差异而改动业务、人员、或积分账期规则。

## 人员规则

正式人员来源：

```sql
dwd.dwd_d_mss_hr_dim_staff_zw10_zw zw
dwd.dwd_d_mss_hr_emp_employee_ext_month emp
dwd.dwd_d_mss_hr_wb_ps_msg_zw wb
```

日账期 `T` 人员分区：

```sql
zw.month_id = M
AND zw.p_acct_day = T
AND emp."month" = M
AND emp.p_acct_day = T
AND wb.month_id = M_MINUS_1
AND wb.p_acct_day = T
```

正式人员过滤：

```sql
emp.ext1 = '1'
AND emp.is_confirm = 1
AND zw.is_off = '1'
AND zw.attr_value IN ('10', '20', '50', '72', '73')
AND wb.code IS NOT NULL
```

实习人员过滤：

```sql
emp.ext1 = '2'
AND emp."month" = M
AND emp.p_acct_day = T
AND emp.intern_start_time <= INTERN_CUTOFF
```

人数统计：

```text
total_people_all：含班长
total_people_eff：不含班长，即正式人员排除 attr_value='20'
实习人员全部计入 total_people_all 和 total_people_eff
```

注意：

- 人员模块不应取最新分区。
- 人员随分区会变动，使用最新分区可能导致历史账期人数变化。

## 不要这样做

不要简单使用：

```sql
MAX(p_acct_day)
```

原因：

- 业务表后续分区会累积或修正部分指标，导致 151、FTTR-B 等全量偏离日账期。
- 积分是 `T+1` 分区落表，账期和分区不是同一天。
- 人员分区取最新会导致历史账期人数变化。

也不要把文件名中的导出日期直接作为所有模块的 `p_acct_day`。

## 已验证样例

以 `T=20260702` 为例：

| 模块 | 应取口径 | 已验证结果 |
|---|---|---|
| 业务发展量 | `new_date='2026-07-02'`，`p_acct_day='20260702'` | 全量业务可对齐 |
| 积分 | `acct_day='20260702'`，`p_acct_day='20260703'`，明细单条取整后汇总 | 全省 `fzPoint=47101`、`yyPoint=15812`、`sumPoint=62913` |
| 人员 | `month_id/month='202607'`，`p_acct_day='20260702'` | 全省 `total_people_all=2867`、`total_people_eff=2712` |

### 积分规则补充

`20260709_各地市随销统计.xlsx` 与 0702 下半表不同，本文件更接近：

```sql
pt.acct_day = T
AND pt.p_acct_day = T
SUM(TRY_CAST(pt.fz_point AS DOUBLE))
SUM(TRY_CAST(pt.yy_point AS DOUBLE))
```

不要在这个文件上使用：

```sql
SUM(ROUND(pt.fz_point, 0))
SUM(ROUND(pt.yy_point, 0))
```

否则全省会出现：

```text
发展积分少 3.4
运营积分少 4.8
原始/价值积分少 8.2
```

结论：

```text
score=T 文件优先按 T 分区核验；
积分是否 ROUND 要看 Excel 展示口径，不同日期/版本可能不同。
```

### 人数规则补充

当前 SQL 使用：

```sql
emp.ext1 = '2'
AND emp."month" = M
AND emp.p_acct_day = T
AND emp.intern_start_time <= date_format(date_add('month', -3, date_parse(T, '%Y%m%d')), '%Y-%m-%d')
```

即 `T=20260709` 时 cutoff 为：

```text
2026-04-09
```

但 `20260709_各地市随销统计.xlsx` 的总人数不符合该 cutoff。当前 SQL 人数：

```text
正式人员 2829
实习人员 22
合计 2851
```

Excel 总人数：

```text
2961
```

实习人员放宽到：

```sql
emp.intern_start_time <= '2026-06-09'
```

后变为：

```text
正式人员 2829
实习人员 138
合计 2967
```

与 Excel 只差 6。说明该 Excel 的总人数更接近“纳入 6 月 9 日前实习人员”的口径，而不是简单 `T-3个月`。

后续继续验证“实习一年内”：

```text
一年内到 T=2026-07-09：正式+实习=3055，比 Excel 2961 多 94。
一年内到月初 2026-07-01：正式+实习=3036，比 Excel 2961 多 75。
```

因此 20260709 文件不应使用“实习一年内”作为总人数口径。

当前最贴近全省总人数的候选口径是：

```sql
emp.ext1 = '2'
AND emp."month" = '202607'
AND emp.p_acct_day = '20260709'
AND emp.is_confirm = 0
AND emp.intern_start_time <= '2026-06-20'
```

该候选口径下：

```text
正式人员 2829
实习人员 132
合计 2961
```

全省总人数与 Excel 一致，但地市仍有小差异，暂作为待确认候选规则，不替代通用实习人员规则。

后续按业务提示验证了另一条更合理的实习候选规则：

```text
实习人员 = emp.ext1='2' 且 zw_type='1'（新装维）
```

对应 20260709：

```sql
emp.ext1 = '2'
AND emp."month" = '202607'
AND emp.p_acct_day = '20260709'
AND emp.intern_start_time <= '2026-06-09'
AND e.zw_type = '1'
```

其中：

```text
2026-06-09 = T 往前 1 个月
地市归属用 emp.ext7
人员唯一键用 emp.ext18
zw_type='1' 表示新装维
```

验证结果：

| 口径 | 正式人数 | 实习人数 | 合计 | Excel | 差异 |
|---|---:|---:|---:|---:|---:|
| 当前 SQL，实习 cutoff=2026-04-09 | 2829 | 22 | 2851 | 2961 | -110 |
| 实习一年内到 2026-07-09 | 2829 | 226 | 3055 | 2961 | 94 |
| `ext1=2` + `zw_type=1` + `intern_start_time<=2026-06-09` | 2829 | 138 | 2967 | 2961 | 6 |

| 地市 | SQL-like | Excel | SQL-Excel |

| 威海 | 98 | 95 | 3 |
| 淄博 | 120 | 119 | 1 |
| 临沂 | 358 | 357 | 1 |
| 东营 | 82 | 81 | 1 |

结论：

```text
20260709 人数口径应优先继续沿 ext1=2 + zw_type=新装维 + 实习一个月 方向排查；
“一年内”已验证会查多，不应采用。
```

进一步排查后，对 20260709 可固化为当前候选规则：

```sql

AND emp."month" = '202607'
AND emp.p_acct_day = '20260709'
```
